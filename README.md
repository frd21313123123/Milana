# Telegram-клиент на Python

CLI-программа подключается к обычному пользовательскому аккаунту Telegram через
MTProto и библиотеку Telethon. Она умеет показывать диалоги, читать историю,
отправлять текстовые сообщения, выводить новые сообщения в реальном времени и
запускать ИИ-ответчик для одного разрешённого пользователя.

> Используйте программу только со своим аккаунтом. Не публикуйте `.env` и файл
> `sessions/telegram_account.session`: файл сессии даёт доступ к аккаунту.

## Установка

Требуется Python 3.10 или новее.

```powershell
cd D:\python-w1\AI-goi
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -r requirements.txt
```

Данные `api_id` и `api_hash` со скриншота уже записаны в локальный `.env`.
Файл исключён из Git.

## Первый вход

```powershell
python telegram_client.py login
```

Telethon запросит номер телефона в международном формате, код из Telegram и,
если включена двухэтапная аутентификация, облачный пароль. После входа рядом со
скриптом появится защищаемый файл `sessions/telegram_account.session`, и повторно
вводить код обычно не потребуется.

## Команды

```powershell
# Показать подключённый аккаунт
python telegram_client.py me

# Последние 30 диалогов и их ID
python telegram_client.py dialogs --limit 30

# Последние 20 сообщений (username или числовой ID из команды dialogs)
python telegram_client.py read @username --limit 20
python telegram_client.py read -1001234567890 --limit 20

# Отправить сообщение; текст с пробелами можно передать без кавычек
python telegram_client.py send @username Привет! Это тест.

# Слушать новые сообщения во всех диалогах или только в одном
python telegram_client.py listen
python telegram_client.py listen @username

# Запустить ИИ-ответчик для всех входящих сообщений
python telegram_client.py ai-bot

# Показать текущее состояние и расписание Миланы
python telegram_client.py schedule
python telegram_client.py schedule --brief
python telegram_client.py schedule --day mon
```

## ИИ-ответчик

Команда `ai-bot` реагирует на все входящие текстовые сообщения и фото. Каждому
чату принадлежит отдельная очередь и один worker, поэтому разные чаты ожидают и
генерируют ответы независимо. Первое сообщение остаётся непрочитанным до
естественного момента ответа по расписанию. Затем клиент ждёт две секунды тишины
(но не больше восьми дополнительных секунд) и объединяет все накопившиеся
сообщения этого чата в один пакет. Отметка о прочтении ставится сразу до
последнего сообщения пакета, а первая часть ответа привязывается именно к нему.

Если новая реплика приходит, пока модель формирует ответ, устаревший черновик не
отправляется: генерация запускается заново со всем накопленным пакетом. Милана
сама выбирает, ответить одним сообщением или естественной серией до пяти
сообщений. Между частями выдерживается короткая пауза с индикатором
«печатает…». Если пользователь пишет во время этой паузы, неотправленный остаток
серии отменяется, а новое сообщение становится началом продолжения диалога.

Исходящие сообщения самого аккаунта игнорируются, чтобы не возникал цикл
ответов. Telegram-фото, а также
JPEG, PNG, WebP и GIF, отправленные как файл, передаются модели вместе с подписью;
фото без подписи тоже обрабатываются. Остальные медиа без текста проходят ту же
задержку чтения, но модели не передаются.

### Память и дневник

ИИ-ответчик хранит память локально в `data/milana_memory.sqlite3`. Для каждого
Telegram-чата сохраняется отдельная история: сообщения одного собеседника не
попадают в контекст другого. При первом сообщении после подключения импортируется
до 40 предыдущих текстовых сообщений этого чата, затем перед каждым ответом
модели передаётся свежий хвост локальной истории. Сообщения активного входящего
пакета добавляются к нему отдельно и ровно по одному разу. Каждая реально
отправленная часть ответа сохраняется отдельной строкой с настоящим Telegram ID,
поэтому прерванная серия и частичная ошибка отправки не создают в памяти текст,
которого собеседник не видел.

Дневник Миланы общий для всех чатов. Модель сама вызывает функцию
`write_diary`, когда решает сохранить устойчивый факт, предпочтение, обещание
или важное событие. Записи дневника доступны ей в любом чате и сохраняются после
перезапуска программы. Служебный вызов функции пользователю не отправляется.
Промпт запрещает сохранять пароли, платёжные данные и одноразовые коды, однако
дневник всё равно является локальным чувствительным файлом — не публикуйте его и
ограничьте доступ к каталогу проекта. Каталог `data/` исключён из Git.

Активные очереди и ещё не отправленные черновики хранятся только в оперативной
памяти. После остановки процесса они не возобновляются как отдельные задания.
Существующий импорт Telegram-истории восстановит их как контекст при следующем
входящем сообщении, но не гарантирует самостоятельный ответ без нового события.

Во время сна Милана обычно не читает сообщения, не показывает «печатает…» и не
отвечает. Исключение — активная переписка, продолжившаяся после отбоя в пределах
30 минут от последнего ответа Миланы. Остальные ночные сообщения не
отбрасываются: при непрерывной работе бота они будут обработаны после пробуждения
и небольшой утренней задержки. Если за время ожидания занятие сменилось, момент
ответа пересчитывается уже по новому занятию.

Во время обычного ожидания аккаунт остаётся «не в сети». Когда ответ готов к
отправке, Милана появляется онлайн. Если новое сообщение пришло, пока она уже
показывается «в сети», сначала запускается короткий таймер (1–10 секунд), и только
после его окончания Милана читает сообщение (ставит отметку о прочтении), генерирует
ответ и отправляет его. После каждого успешно отправленного ответа статус гарантированно
«в сети» ещё случайные 30–60 секунд. Вне активной переписки новый ответ не
начинается, если до отбоя осталось меньше минуты. При штатной остановке статус
переключается на «не в сети». Видимость статуса для конкретных людей также
зависит от настроек приватности «Последняя активность» в Telegram.

Добавьте в локальный `.env` API-ключ OpenAI (не публикуйте этот файл):

```dotenv
OPENAI_API_KEY=
```

Настройки модели находятся в `ai_config.json` рядом со скриптом:

```json
{
  "model": "gpt-5.6-terra",
  "system_prompt": "Ты — Милана. Отвечай дружелюбно и кратко.",
  "temperature": 0.7,
  "max_output_tokens": 1200,
  "message_flow": {
    "input_quiet_seconds": 2.0,
    "input_max_wait_seconds": 8.0,
    "max_reply_messages": 5,
    "inter_message_min_delay_seconds": 1.0,
    "inter_message_max_delay_seconds": 3.0
  }
}
```

- `model` — идентификатор модели, доступной вашему API-проекту;
- `system_prompt` — системный промпт; он может быть многострочным;
- `temperature` — вариативность ответов, от `0` до `2`; если выбранная модель
  не поддерживает параметр, клиент автоматически повторяет запрос без него;
- `max_output_tokens` — максимальная длина ответа, от `1` до `4000`.
- `message_flow.input_quiet_seconds` — сколько секунд тишины ждать после последней
  входящей реплики;
- `message_flow.input_max_wait_seconds` — максимальное дополнительное ожидание
  накопления пакета;
- `message_flow.max_reply_messages` — защитный предел смысловых сообщений в одном
  ответе (`1` отключает смысловое разбиение);
- `message_flow.inter_message_min_delay_seconds` и
  `inter_message_max_delay_seconds` — диапазон пауз между частями ответа.

Каждую смысловую часть клиент дополнительно делит по техническому лимиту Telegram
в 4000 символов. Такое деление не влияет на решение Миланы о структуре ответа.

Измените файл и перезапустите `ai-bot`. Если `ai_config.json` отсутствует,
клиент использует прежние переменные `.env`: `OPENAI_MODEL` и
`AI_SYSTEM_PROMPT`, либо встроенные значения.

## Расписание Миланы

Расписание хранится в `milana_schedule.json` и по умолчанию повторяет активный
студенческий пресет приложенного симулятора:

- будни: подъём в 07:30, отбой в 23:30, шесть часов учёбы, дорога, перерывы,
  прогулка, спорт и личное время;
- выходные: подъём в 09:30, отбой в 00:30, длинная прогулка, спорт и личное
  время;
- часовой пояс: `Asia/Yekaterinburg` (`UTC+05:00`).

Движок строит каждый день в той же последовательности, что HTML-шаблон, и по
тем же формулам считает энергию, стресс, продуктивность и баланс. Текущее
занятие, следующее занятие и показатели дня добавляются в инструкции модели
перед каждым ответом. Поэтому Милана отвечает на вопросы о том, что она сейчас
делает, с учётом реального времени.

Задержки чтения и ответа тоже зависят от текущего занятия:

| Занятие | Когда Милана читает и отвечает |
|---|---|
| Сон | Не читает и не отвечает |
| Утренние сборы, завтрак | Через 15 секунд–2 минуты |
| Дорога | Через 15 секунд–3 минуты |
| Дорога домой | Через 30 секунд–5 минут |
| Учёба или работа | Через 4–24 минуты |
| Перерыв или приём пищи | Через 5 секунд–2 минуты |
| Прогулка | Через 45 секунд–6 минут |
| Спорт | Через 1–8 минут |
| Личные дела | Через 1–10 минут |
| Свободное время | Через 2 секунды–1 минуту |

Конкретный момент внутри диапазона выбирается случайно. Если раньше наступает
следующее занятие, старый таймер отменяется и диапазон выбирается заново для
нового состояния. Это не даёт длинной задержке с учёбы переноситься на уже
начавшийся перерыв и не позволяет ответу уйти во время сна.

Все параметры находятся в `milana_schedule.json`:

- `weekday` и `weekend` задают подъём, отбой, продолжительность учёбы/работы,
  прогулки, личных дел, дороги и перерывов;
- `custom_events` добавляет или перекрывает занятие в конкретный день и время;
- `response_behavior.default` задаёт общую задержку ответа;
- `response_behavior.by_kind` задаёт задержки по типу занятия;
- `response_behavior.by_title` задаёт более приоритетные правила по точному
  названию занятия;
- `online_behavior` управляет быстрым ответом и статусом «в сети».

Например, правила онлайна можно менять прямо в этом блоке:

```json
"online_behavior": {
  "online_response_min_seconds": 1,
  "online_response_max_seconds": 10,
  "post_reply_online_min_seconds": 30,
  "post_reply_online_max_seconds": 60,
  "spontaneous_online_interval_min_seconds": 900,
  "spontaneous_online_interval_max_seconds": 2700,
  "spontaneous_online_duration_min_seconds": 120,
  "spontaneous_online_duration_max_seconds": 300,
  "sleep_buffer_seconds": 60,
  "conversation_sleep_delay_seconds": 1800
}
```

По умолчанию в доступное по расписанию время Милана заходит в сеть через
случайные 15–45 минут, остаётся там на 2–5 минут и снова выходит. Во время сна
такие фоновые появления не запускаются.

Минимальное значение не должно превышать максимальное, а
`sleep_buffer_seconds` должно быть не меньше
`post_reply_online_max_seconds`, иначе гарантированное online-окно может попасть
на сон.

Для статуса «в сети» диапазоны намеренно ограничены: ответ планируется максимум
через 10 секунд, а после отправки ответа аккаунт остаётся online от 30 до 60
секунд. Если сообщение приходит, пока Милана в сети, сначала проходит случайная короткая
задержка, и только затем сообщение отмечается прочитанным и начинается генерация ответа.

После каждого отправленного ответа Милана откладывает сон на
`conversation_sleep_delay_seconds` (по умолчанию 30 минут). Если плановый отбой
уже наступил, входящее сообщение, полученное до конца этого окна, продолжает
переписку; новый ответ заново отсчитывает полные 30 минут.

Параметры будней и выходных можно менять прямо в `milana_schedule.json`.
Дополнительные события добавляются в нужный день (`mon`–`sun`), например:

```json
"wed": [
  {
    "title": "Бассейн",
    "type": "sport",
    "start": "18:00",
    "end": "19:00"
  }
]
```

Пользовательское событие имеет приоритет при определении текущего занятия.
Секция `response_behavior` в том же JSON задаёт общий диапазон, диапазоны по
типу занятия (`by_kind`) и более точные исключения по названию (`by_title`).
Правило по названию имеет приоритет, поэтому, например, «Дорога домой» может
отличаться от обычной дороги. Пользовательские события автоматически получают
правило своего `type`. После изменения файла перезапустите ИИ-ответчик, чтобы он
загрузил новое расписание и правила ответа.

## Запуск через BAT-файл

Для обычного запуска используйте `bot_control.bat`: двойной клик открывает меню.
Из терминала доступны команды:

```powershell
.\bot_control.bat start
.\bot_control.bat stop
.\bot_control.bat status
.\bot_control.bat logs
```

Запуск выполняется скрыто. Идентификатор процесса сохраняется в `bot.pid`, а
последний вывод доступен в `bot-output.log` и `bot-error.log`. Команда `stop`
ищет все локальные процессы Миланы (`telegram_client.py ai-bot`) и завершает
их, в том числе запущенные вручную, а не только через BAT-файл.

Меню BAT-файла и команды `start`, `stop`, `status` показывают отдельной строкой
текущее занятие Миланы, его временной интервал, оставшееся время и следующее
занятие, а также энергию, стресс, продуктивность и баланс текущего дня.

Для проверки отправки без беспокойства других людей используйте адресата `me`:

```powershell
python telegram_client.py send me Проверка подключения
python telegram_client.py read me --limit 5
```

Справка доступна через `python telegram_client.py --help` и через `--help` после
любой команды.
